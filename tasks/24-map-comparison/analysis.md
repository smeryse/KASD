# Анализ эффективности MyHashMap vs MyTreeMap

## Выводы

### Операция PUT (добавление элементов)
- **MyHashMap**: O(1) амортизированное время добавления.
  - Вычисление хеш-кода: O(1)
  - Определение бакета: O(1)
  - Добавление в начало списка: O(1)
  - Resize: O(n), но происходит редко (амортизация)
- **MyTreeMap**: O(log n) время добавления.
  - Обход дерева: O(log n)
  - Создание нового узла: O(1)
- **Вывод**: MyHashMap значительно быстрее при добавлении, особенно на больших размерах.

### Операция GET (получение по ключу)
- **MyHashMap**: O(1) среднее время.
  - Вычисление хеша: O(1)
  - Поиск в бакете: O(1) при хорошем хешировании
  - В худшем случае (коллизии): O(n)
- **MyTreeMap**: O(log n) время.
  - Обход бинарного дерева: O(log n)
  - Сравнение ключей на каждом узле
- **Вывод**: MyHashMap быстрее, но разрыв меньше, чем у PUT.

### Операция REMOVE (удаление по ключу)
- **MyHashMap**: O(1) среднее время.
  - Поиск элемента: O(1)
  - Удаление из списка: O(1)
- **MyTreeMap**: O(log n) время.
  - Поиск элемента: O(log n)
  - Удаление узла: O(log n) (перебалансировка)
- **Вывод**: MyHashMap быстрее, разница пропорциональна размеру данных.

## Общее заключение
1. **MyHashMap** эффективнее для всех операций в среднем случае.
2. **MyTreeMap** обеспечивает предсказуемую производительность O(log n).
3. **MyHashMap** не сохраняет порядок ключей, **MyTreeMap** хранит ключи отсортированными.
4. При необходимости навигации (lowerKey, higherKey, range queries) **MyTreeMap** незаменим.
5. Для простого хранения и быстрого доступа **MyHashMap** — лучший выбор.
