# Инструкция к лабораторной работе №9
## DJ Music Library — GUI-приложение

---

## 1. Что было сделано

Создано десктопное приложение для работы с базой данных **DJ Music Library**.
Приложение позволяет просматривать, добавлять, редактировать и удалять записи из БД PostgreSQL, а также просматривать отчёты на основе представлений (views).

**Технологии:**
- Язык: C# (.NET 8)
- UI-фреймворк: Avalonia UI (кроссплатформенный аналог WPF)
- База данных: PostgreSQL
- Драйвер БД: Npgsql

## 4. Функционал приложения

### 4.1. Меню «Таблицы»
Содержит 5 таблиц БД:
- **Artist** — артисты (1925 записей)
- **Genre** — жанры (24 записи)
- **Track** — треки (2790 записей)
- **Collection** — коллекции/сеты (13 записей)
- **Event** — события/мероприятия (5 записей)

При выборе таблицы данные загружаются и отображаются в списке.

### 4.2. Меню «Представления»
Содержит 4 представления (views) — это сохранённые SQL-запросы, которые объединяют данные из нескольких таблиц:

| Представление | Описание |
|---|---|
| **Списки треков** (v_track_lists) | Треки с именами артистов и жанров |
| **Подготовленные сеты** (v_prepared_sets) | Коллекции с перечнем треков |
| **История выступлений** (v_performance_history) | События с информацией о сетах |
| **Отчёт менеджера событий** (v_event_manager_report) | Сводный отчёт по мероприятиям |

Представления доступны **только для чтения** — кнопки Добавить/Редактировать/Удалить заблокированы.

### 4.3. Панель инструментов
- **Добавить** — открывает форму для создания новой записи
- **Редактировать** — открывает форму с данными выбранной записи
- **Удалить** — удаляет выбранную запись (с подтверждением)
- **Обновить** — перезагружает данные из БД

### 4.4. Поиск
Поле ввода + кнопки «Найти» / «Сброс». Ищет по всем столбцам (регистронезависимо). Показывает количество найденных записей.

### 4.5. Сортировка
Клик на заголовок столбца:
- 1-й клик → сортировка по возрастанию (▲)
- 2-й клик → по убыванию (▼)
- 3-й клик → сброс сортировки

Числовые столбцы сортируются как числа, текстовые — по алфавиту.

### 4.6. Пагинация
По 25 записей на страницу. Кнопки «← Назад» и «Вперёд →» для навигации.

## 5. Разбор кода (для презентации)

### 5.1. DbHelper.cs — работа с базой данных

Это статический класс-помощник. Содержит:

| Метод | Что делает |
|---|---|
| `ExecuteQuery(sql)` | Выполняет SELECT, возвращает DataTable |
| `ExecuteNonQuery(sql)` | Выполняет INSERT/UPDATE/DELETE, возвращает кол-во затронутых строк |
| `GetPrimaryKey(table)` | Определяет имя первичного ключа таблицы через системные таблицы PostgreSQL |
| `GetTableNames()` | Возвращает список таблиц |
| `GetViewNames()` | Возвращает список представлений |

**Строка подключения:**
```csharp
Host=localhost;Port=5432;Database=dj_db;Username=postgres;Password=19022016
```

### 5.2. FieldMapper.cs — перевод на русский

Содержит словарь соответствий: `имя_таблицы → { имя_колонки → русское_название }`.

Используется в двух местах:
1. Заголовки столбцов в таблице отображаются на русском
2. Метки полей в форме редактирования — тоже на русском

Метод `IsPrimaryKey()` определяет, является ли поле первичным ключом. Первичные ключи **скрыты** в формах добавления/редактирования (пользователь не должен их менять).

### 5.3. MainWindow.axaml.cs — главное окно

**Ключевые поля:**
```csharp
_allRows       — все строки из БД
_filteredRows  — строки после фильтрации (поиск)
_sortedRows    — строки после сортировки
_currentPage   — текущая страница пагинации
_sortColumn    — индекс столбца для сортировки
_sortDirection — направление сортировки
```

**Метод LoadData():**
1. Выполняет SQL-запрос `SELECT * FROM table`
2. Сохраняет все строки в `_allRows`
3. Вызывает `ApplyFilterAndSort()` — применяет поиск и сортировку
4. Вызывает `BuildList()` — строит UI-элементы
5. Вызывает `UpdatePagination()` — обновляет кнопки пагинации

**Метод ApplyFilterAndSort():**
1. Если есть текст поиска — фильтрует `_allRows` по всем столбцам
2. Если выбрана сортировка — сортирует результат
3. Определяет тип столбца (числовой/текстовый) для правильной сортировки

**Метод BuildList():**
1. Создаёт строку заголовков с индикаторами сортировки (▲/▼)
2. Создаёт строки данных (по 25 на страницу)
3. Каждый заголовок кликабельный — при клике вызывается `Header_Click()`

### 5.4. EditWindow.axaml.cs — окно редактирования

**Конструктор:**
1. Запрашивает схему таблицы (`SELECT * LIMIT 0`)
2. Пропускает поле первичного ключа (не показывает пользователю)
3. Для каждого поля создаёт Label + TextBox
4. Если редактирование — заполняет TextBox текущими значениями

**Метод Btn_Save_Click():**
1. Собирает значения из всех TextBox
2. Проверяет обязательные поля (не nullable)
3. Если добавление — генерирует `INSERT INTO ...`
4. Если редактирование — генерирует `UPDATE ... SET ... WHERE pk = id`
5. Устанавливает `Saved = true` и закрывает окно

**Обработка ошибок FK:**
При удалении записи, на которую ссылаются другие таблицы, ловится исключение `23503` и показывается понятное сообщение.

## 6. Структура базы данных

### Таблицы

| Таблица | Первичный ключ | Поля |
|---------|---------------|------|
| artist | artist_id | name, country, style, active_years, bio |
| genre | genre_id | name, parent_genre_id, bpm_range, description |
| track | track_id | title, artist_id, genre_id, bpm, key, duration, file_format, file_path, rating, play_count, date_added, comments |
| collection | collection_id | name, type, description, style, planned_duration, created_at, notes, total_duration |
| collectiontrack | (collection_id, track_id) | position, transition_notes |
| event | event_id | venue, city, date, audience_size, event_type, collection_id, feedback, earnings |

### Представления (Views)

**v_track_lists** — объединяет track + artist + genre:
```sql
SELECT t.track_id, t.title, a.name, g.name, t.bpm, t.key, t.duration, t.rating, t.play_count
FROM track t JOIN artist a ON t.artist_id = a.artist_id JOIN genre g ON t.genre_id = g.genre_id
```

**v_prepared_sets** — коллекции с треками:
```sql
SELECT c.collection_id, c.name, c.type, c.style, array_agg(t.title), count(*), sum(t.duration)
FROM collection c LEFT JOIN collectiontrack ct ON c.collection_id = ct.collection_id LEFT JOIN track t ON ct.track_id = t.track_id
GROUP BY c.collection_id
```

**v_performance_history** — события с сетами:
```sql
SELECT e.event_id, e.venue, e.city, e.date, e.event_type, e.audience_size, c.name, e.feedback, e.earnings
FROM event e LEFT JOIN collection c ON e.collection_id = c.collection_id
```

**v_event_manager_report** — сводный отчёт:
```sql
SELECT e.event_id, e.venue, e.city, e.date, e.event_type, e.audience_size, c.name,
       count(ct.track_id), round(sum(t.duration)/60.0, 2), round(avg(t.rating), 2), e.earnings,
       CASE WHEN e.audience_size >= 10000 THEN 'Крупное' ... END
FROM event e LEFT JOIN collection c ... LEFT JOIN collectiontrack ct ... LEFT JOIN track t ...
GROUP BY e.event_id
```

## 8. Возможные вопросы от препода и ответы

**В: Почему Avalonia, а не WPF?**
О: WPF работает только на Windows. Avalonia — кроссплатформенный аналог с таким же XAML-синтаксисом, работает на Linux.

**В: Почему не DataGrid, а ListBox?**
О: Avalonia DataGrid имеет проблемы с динамической привязкой данных. ListBox с динамически создаваемыми элементами работает стабильнее.

**В: Как работает сортировка?**
О: При клике на заголовок определяется тип столбца (числовой/текстовый). Список сортируется в памяти с помощью `List.Sort()` с компаратором.

**В: Как работает поиск?**
О: Перебираются все строки и все столбцы. Если значение столбца содержит искомый текст (без учёта регистра), строка добавляется в результат.

**В: Почему поле ID скрыто в форме?**
О: Первичный ключ генерируется автоматически (SERIAL / GENERATED ALWAYS AS IDENTITY). Пользователь не должен его указывать.

**В: Что такое представления (views)?**
О: Это сохранённые SQL-запросы, которые выглядят как таблицы, но данные в них формируются динамически из нескольких таблиц при каждом обращении.
