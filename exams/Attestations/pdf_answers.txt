•

Поиск минимума в неотсортированном массиве (время
работы)
По лекции для алгоритма поиска минимума получается линейное время: 𝑇(𝑛) = 2𝑛 ⇒ 𝑂(𝑛).

csharp
static int FindMin(int[] arr)
{

for (int i = 1; i < arr.Length; i++)
{

Почему нижняя оценка для задачи — 𝛺(𝑛)?
Потому что для корректного поиска минимума в неотсортированном массиве алгоритм обязан

2. Опишите алгоритм сортировки вставками. Какова его временная сложность в
худшем, лучшем и среднем случае? Что такое «инверсия в массиве»? Опишите
модификацию сортировки слиянием, позволяющую подсчитать количество инверсий
за (   log n n )

Сортировка вставками
Алгоритм: Массив условно делится на отсортированную и неотсортированную части. Каждый
следующий элемент вставляется на своё место в отсортированной части массива путём сдвига
элементов вправо.
Временная сложность:
•

лучший случай (массив отсортирован): O(n)

•

средний случай: O(n²)

•

худший случай (обратный порядок): O(n²)

Код сортировки вставками (C#):

csharp
static void InsertionSort(int[] a)
{
for (int i = 1; i < a.Length; i++)
{
int key = a[i];
int j = i - 1;
while (j >= 0 && a[j] > key)
{
a[j + 1] = a[j];
j--;
}
a[j + 1] = key;
}
}

Инверсия в массиве
Инверсия — это пара индексов (i < j), для которых выполняется (a[i] > a[j]). Количество инверсий
показывает степень неупорядоченности массива.

Подсчёт инверсий за O(n·log n) (модификация сортировки слиянием)

Идея: Используется сортировка слиянием. При слиянии двух отсортированных подмассивов, если
элемент из правой части меньше элемента из левой, то он образует инверсии со всеми
оставшимися элементами левой части.
Сложность алгоритма: O(n·log n).
Код подсчёта инверсий (C#):

csharp
static long CountInversions(int[] a)
{
int[] temp = new int[a.Length];
return MergeSort(a, temp, 0, a.Length - 1);
}
static long MergeSort(int[] a, int[] temp, int left, int right)
{
long invCount = 0;
if (left < right)
{
int mid = (left + right) / 2;
invCount += MergeSort(a, temp, left, mid);
invCount += MergeSort(a, temp, mid + 1, right);
invCount += Merge(a, temp, left, mid, right);
}
return invCount;
}
static long Merge(int[] a, int[] temp, int left, int mid, int right)
{
int i = left, j = mid + 1, k = left;
long invCount = 0;
while (i <= mid && j <= right)
{
if (a[i] <= a[j])
temp[k++] = a[i++];
else
{
temp[k++] = a[j++];
invCount += (mid - i + 1);
}
}
while (i <= mid)
temp[k++] = a[i++];
while (j <= right)
temp[k++] = a[j++];
for (int p = left; p <= right; p++)
a[p] = temp[p];
return invCount;
}

3. Опишите алгоритм сортировки слиянием. Почему его сложность всегда (   log n n
)?
Сортировка слиянием
Алгоритм: Массив рекурсивно делится на две части до подмассивов длины 1. Затем подмассивы
попарно сливаются в отсортированном порядке.
Этапы:
1.

Разделить массив на две половины.

2.

Отсортировать каждую половину рекурсивно.

3.

Слить две отсортированные половины в один массив.

Почему сложность всегда O(n·log n)
•

Глубина рекурсии равна log n (массив каждый раз делится пополам).

•

На каждом уровне рекурсии выполняется слияние всех элементов, что требует O(n)
операций.

•

Итоговая сложность: O(n·log n) независимо от входных данных.

Код сортировки слиянием (C#)

csharp
static void MergeSort(int[] a)
{

static void MergeSortRec(int[] a, int[] temp, int left, int right)
{
if (left < right)

MergeSortRec(a, temp, left, mid);
MergeSortRec(a, temp, mid + 1, right);
Merge(a, temp, left, mid, right);
}
}

temp[k++] = a[j++];

4. Что такое «очередь с приоритетом»? Какие основные операции она должна
поддерживать? Опишите структуру данных «бинарная куча» для минимума. Какое
свойство она должна поддерживать?
Очередь с приоритетом — АТД, который хранит мультимножество сравнимых элементов и
поддерживает основные операции:
•

insert(x) — добавить элемент

•

get_min() (или find_min/peek) — вернуть минимальный элемент

•

remove_min() (или extract_min) — удалить минимальный элемент (иногда extract_min
совмещает «вернуть + удалить»)

Бинарная куча (min-heap) как реализация:
•

Это полное бинарное дерево (последний уровень заполняется слева направо).

•

Хранится в массиве: корень a[0].
–

left(i) = 2*i + 1

–

right(i) = 2*i + 2

–

parent(i) = (i - 1) / 2 (целочисленно)

Свойство (для минимума): значение в каждом узле ≤ значений в его дочерних узлах; следствие —
минимум всегда в корне.
Операции очереди с приоритетом (min): insert(x), get_min(), remove_min() (иногда
объединяют в extract_min). filecite turn1file12
Бинарная куча (min-heap): полное бинарное дерево, хранится в массиве:
left(i)=2*i+1, right(i)=2*i+2, parent(i)=(i-1)//2. Свойство: в каждом узле значение ≤
значений в детях, поэтому минимум в корне. filecite turn1file17

Реализация на C# (heap_get_min, heap_insert, heap_remove_min, sift_up,
sift_down)
csharp
using System;
using System.Collections.Generic;

public class MinHeap
{
private readonly List<int> a = new List<int>();
public int Count => a.Count;
public int GetMin()
{
if (a.Count == 0) throw new InvalidOperationException("Куча пуста");
return a[0];
}
public void Insert(int x)
{
a.Add(x);
SiftUp(a.Count - 1);
}
public int RemoveMin()
{
if (a.Count == 0) throw new InvalidOperationException("Куча пуста");
int minVal = a[0];
int lastVal = a[a.Count - 1];
a.RemoveAt(a.Count - 1);
if (a.Count > 0)
{
a[0] = lastVal;
SiftDown(0);
}
return minVal;
}
private void SiftUp(int idx)
{
while (idx > 0)
{
int parentIdx = (idx - 1) / 2;
if (a[idx] < a[parentIdx])
{
(a[idx], a[parentIdx]) = (a[parentIdx], a[idx]);
idx = parentIdx;
}
else
{
break;
}
}
}
private void SiftDown(int idx)
{
int n = a.Count;

while (idx * 2 + 1 < n)
{
int leftIdx = 2 * idx + 1;
int rightIdx = 2 * idx + 2;
int minChildIdx = leftIdx;
if (rightIdx < n && a[rightIdx] < a[leftIdx])
minChildIdx = rightIdx;
if (a[idx] <= a[minChildIdx])
break;
(a[idx], a[minChildIdx]) = (a[minChildIdx], a[idx]);
idx = minChildIdx;
}
}
}

5. Объясните алгоритмы операций вставка и извлечения минимума в бинарной куче.
Почему их время работы – (  logn ) ? Как работает алгоритм построения кучи из
неупорядоченного массива? Почему его сложность – ( )  n , а не (   log n n ) ?

Вставка в min-heap (insert) и почему O(log n)
Алгоритм:
1.

Добавляем новый элемент x в конец массива (в последнюю свободную позицию).

2.

Делаем sift_up (просеивание вверх): сравниваем с родителем, если меньше — меняем
местами, повторяем до корня или пока порядок не восстановится.

Псевдокод:

heap_insert(a, x)
a.append(x)
sift_up(a, len(a) - 1)
sift_up(a, idx)
while idx > 0
parent_idx = (idx - 1) // 2
if a[idx] < a[parent_idx]
swap(a[idx], a[parent_idx])
idx = parent_idx
else
break
Почему O(log n): высота дерева ~ log2 n, sift_up поднимает элемент не более чем на высоту
дерева.

Извлечение минимума (remove_min) и почему O(log n)
Алгоритм:
1.

Запоминаем корень (минимум).

2.

Перемещаем последний элемент массива в корень.

3.

Делаем sift_down (просеивание вниз): сравниваем с детьми, если больше — меняем
местами с наименьшим ребёнком, повторяем пока порядок не восстановится или не дошли
до листа.

Псевдокод:

heap_remove_min(a)
if not a: raise Exception("Куча пуста")
min_val = a[0]
last_val = a.pop()
if a:
a[0] = last_val
sift_down(a, 0)
return min_val, a
sift_down(a, idx)
n = len(a)
while idx * 2 + 1 < n
left_idx = 2 * idx + 1
right_idx = 2 * idx + 2
min_child_idx = left_idx
if right_idx < n and a[right_idx] < a[left_idx]
min_child_idx = right_idx
if a[idx] <= a[min_child_idx]
break
else
swap(a[idx], a[min_child_idx])
idx = min_child_idx
Почему O(log n): sift_down опускает элемент не более чем на высоту дерева, то есть O(log n).

Построение кучи из неупорядоченного массива (heapify)
В файле указано 2 способа:
1.

Последовательная вставка: вызвать insert для каждого из n элементов → O(n log n).

2.

Линейный алгоритм: идти снизу вверх и вызывать sift_down для каждого узла, начиная с
последнего нелистового → O(n).

Псевдокод (heapify для max-heap в лекциях):

heapify(arr)
n = len(arr)

start_idx = n // 2 - 1
for i in range(start_idx, -1, -1)
sift_down_max(arr, i, n)

Почему heapify — O(n), а не O(n log n)
Обоснование из файла:
•

На нижнем уровне ~ n/2 узлов, их высота 0 → sift_down = 0 операций

•

На уровне выше ~ n/4 узлов, высота 1 → не более 1 операции

•

Далее: n/8 узлов высоты 2, …, в корне 1 узел высоты h

Суммарное время:

T(n) = 0*(n/2) + 1*(n/4) + 2*(n/8) + 3*(n/16) + ... + h*1
Эта сумма ограничена сверху числом 2n, значит T(n) = O(n).

Пирамидальная сортировка (Heap Sort)
Идея алгоритма:
Используется бинарная куча (обычно max-heap). Сначала из массива строится куча, затем
максимальные элементы по одному извлекаются и помещаются в конец массива.

Алгоритм (псевдокод из файла)
Построение кучи
heapify(a, n)
for i ← n/2 − 1 downto 0
sift_down(a, i, n)
Сортировка
heap_sort(a)
n ← length(a)
heapify(a, n)
for i ← n − 1 downto 1
swap(a[0], a[i])
sift_down(a, 0, i)
Просеивание вниз
sift_down(a, i, size)
while left(i) < size
j ← left(i)
if right(i) < size and a[right(i)] > a[j]
j ← right(i)
if a[i] ≥ a[j]
break

swap(a[i], a[j])
i ← j

Почему сложность O(n·log n)
Обоснование из файла:
1.

Построение кучи — O(n)

2.

Основной цикл выполняется n раз

3.

На каждом шаге вызывается sift_down, работающий за O(log n)

Итоговая сложность:
O(n) + n · O(log n) = O(n · log n)

7. Опишите алгоритм бинарного поиска в отсортированном массиве. Какова его
сложность? Что такое нижняя граница и верхняя граница элемента? Как их
реализовать с помощью бинарного поиска?

Бинарный поиск в отсортированном массиве
Алгоритм:
Работает только для отсортированного массива. На каждом шаге сравнивает искомый элемент с
серединой и отбрасывает половину массива.
Псевдокод бинарного поиска
binary_search(a, x)
l ← 0
r ← n − 1
while l ≤ r
m ← (l + r) div 2
if a[m] = x
return m
else if a[m] < x
l ← m + 1
else
r ← m − 1
return −1
Сложность:
На каждом шаге диапазон уменьшается в 2 раза ⇒ O(log n).

Нижняя и верхняя границы
Нижняя граница (lower bound):
Индекс первого элемента ≥ x.
Псевдокод
lower_bound(a, x)
l ← 0
r ← n
while l < r
m ← (l + r) div 2

if a[m] < x
l ← m + 1
else
r ← m
return l
Верхняя граница (upper bound):
Индекс первого элемента > x.
Псевдокод
upper_bound(a, x)
l ← 0
r ← n
while l < r
m ← (l + r) div 2
if a[m] ≤ x
l ← m + 1
else
r ← m
return l
Сложность lower_bound и upper_bound:
Так же, как у бинарного поиска — O(log n).

8. В чём заключается идея «бинарного поиска по ответу»? Приведите пример задачи,
где он применяется.

Идея «бинарного поиска по ответу»
Суть:
Бинарный поиск применяется не к массиву, а к множеству возможных ответов.
Предполагается, что для ответа существует монотонное предикатное условие:
если для значения x условие истинно, то для всех x' > x (или < x) оно тоже истинно.

Общая схема (псевдокод из лекции)
binary_search_answer(l, r)
while l < r
m ← (l + r) div 2
if check(m) = true
r ← m
else
l ← m + 1
return l
где check(m) — проверка, можно ли получить ответ не хуже m.

Пример задачи (из лекции)
Задача:
Дано n копий и два ксерокса, которые печатают за x и y секунд.
Найти минимальное время, за которое можно сделать n копий.
Идея применения:
•

Ответ — это время

•

Проверка check(t): можно ли за t секунд напечатать ≥ n копий

•

Функция check(t) монотонна ⇒ применим бинарный поиск по ответу

Почему работает
Из файла:
•

множество ответов упорядочено

•

проверка «подходит / не подходит» монотонна

•

бинарный поиск находит минимальный подходящий ответ

Сложность: O(log R · cost(check)), где R — диапазон возможных ответов.

детерминированного алгоритма медиана медиан. Почему его время работы в худшем
случае – ( )n  ?

Рандомизированный быстрый выбор (Quickselect) для k-й порядковой
статистики
Задача: найти k-ю порядковую статистику (элемент, который стоял бы на позиции k (с 0) в
отсортированном массиве).
Идея Quickselect (Хоара): очень похож на быструю сортировку, но рекурсивно обрабатывает
только ту часть, где находится ответ.
1.

выбрать случайный опорный элемент x;

2.

сделать partition на < x и ≥ x;

3.

сравнить k с позицией опорного (или границей) и уйти в нужную часть.

Псевдокод (как в лекциях, в стиле реализации):

quickselect(arr, k, left, right)
if left == right
return arr[left]
pivot_idx = random(left, right)
swap(arr[pivot_idx], arr[right])
pivot_value = arr[right]
p = partition(arr, left, right, pivot_value)
if k == p
return arr[p]
else if k < p
return quickselect(arr, k, left, p - 1)
else

return quickselect(arr, k, p + 1, right)
Средняя (ожидаемая) сложность: анализ аналогичен быстрой сортировке; в среднем на каждом
шаге размер задачи уменьшается примерно вдвое ⇒ O(n) в среднем.

Детерминированный алгоритм “медиана медиан” (BFPRT): идея и почему
худший случай O(n)
Идея: выбирать опорный элемент x не случайно, а детерминированно так, чтобы он
гарантированно делил массив не хуже, чем 3:7.
Шаги алгоритма (из лекций):
1.

разбить массив на группы по 5 (последняя может быть меньше);

2.

найти медиану в каждой группе (для 5 элементов — константно);

3.

рекурсивно найти медиану x среди этих медиан (медиану медиан) — это опорный;

4.

разделить массив вокруг x (partition), получить границу m;

5.

как в Quickselect, рекурсивно продолжить поиск в нужной части по k и m.

Почему время в худшем случае O(n): из свойства медианы медиан следует, что после разбиения
вокруг x большая часть имеет размер не более 7n/10.
Получается рекуррентность:
𝑇(𝑛) ≤ 𝑇 (𝑛/5) + 𝑇(7𝑛/10) + 𝑂(𝑛)
В лекциях доказывается по индукции, что из этой рекуррентности следует T(n)=O(n) (выбор
константы, например 𝑐 ≥ 10𝑎).
Вывод: BFPRT находит k-ю порядковую статистику за O(n) в худшем случае (хотя константа
большая, и на практике часто быстрее Quickselect).

10. Что такое «амортизированная сложность»? В чём её отличие от сложности в
худшем случае? Объясните на примере динамического массива методы
агрегирования, бухгалтерского учёта и потенциалов для амортизационного анализа.

Амортизированная сложность
Определение (из файла):
Амортизированная сложность — это средняя стоимость операции на последовательности
операций, гарантирующая верхнюю оценку на любой префикс этой последовательности, а не на
одну операцию.
Отличие от худшего случая:
•

Худший случай — максимальное время одной операции.

•

Амортизированная сложность — средняя стоимость операции на длинной серии операций,
даже если отдельные операции бывают дорогими.

Пример: динамический массив
Массив увеличивает размер в 2 раза при переполнении.
Операция push_back:
•

обычно — O(1)

•

иногда — O(n) (при перераспределении и копировании)

Метод агрегирования
Идея:
Считаем суммарную стоимость n операций и делим на n.
Для n операций push_back:
•

копирование элементов при расширениях суммарно ≤ 2n

•

всего операций ≤ 3n

Амортизированная стоимость:
3n / n = O(1)

Метод бухгалтерского учёта (accounting)
Идея:
Некоторым операциям назначается завышенная цена, излишек сохраняется и используется для
оплаты дорогих операций. filecite turn6file3
Для push_back:
•

берём цену 3

•

1 — за вставку

•

2 — «откладываются» на будущее копирование

Когда происходит расширение массива, накопленные «деньги» покрывают копирование.
⇒ амортизированная сложность O(1).

Метод потенциалов
Идея:
Вводится потенциальная функция Φ, отражающая «запас работы» структуры данных.
Амортизированная стоимость: 𝑐^ = 𝑐 + 𝛥𝛷
Для динамического массива (из лекций):
Φ = 2·size − capacity
•

при обычной вставке Φ увеличивается немного

•

при перераспределении Φ резко уменьшается и компенсирует большую реальную стоимость

⇒ амортизированная стоимость push_back равна O(1).

Итог
•

Амортизированная сложность ≠ средняя по вероятности

•

Она гарантирует верхнюю оценку на последовательность операций

•

Для динамического массива push_back имеет амортизированную сложность O(1)

•

Идея (из файла):
Очередь реализуется с использованием двух стеков S1 и S2.
•

S1 — стек для добавления элементов

•

S2 — стек для извлечения элементов

Операции

Амортизированный анализ
Из файла:
•

•

Каждый элемент:
–

–

–

Вывод:

Кольцевой буфер
Определение:
Кольцевой буфер — это реализация очереди на массиве, где конец массива логически соединён с
началом.
Используются два индекса:
•

head — начало очереди

•

tail — конец очереди

Как решает проблему наивной реализации
Проблема наивной очереди на массиве:
При извлечении элемента требуется сдвиг всех элементов, что даёт O(n).
Решение кольцевого буфера:
Индексы изменяются по модулю размера массива:
head = (head + 1) mod n
tail = (tail + 1) mod n
Элементы не сдвигаются, все операции выполняются за O(1).

Почему внутренний цикл при этом нужно проходить в обратном порядке?

Задача о рюкзаке (0/1 knapsack)
Формулировка (из файла):
Даны n предметов, каждый с весом w[i] и ценностью v[i], и рюкзак вместимости W.
Нужно выбрать подмножество предметов (каждый можно взять не более одного раза), чтобы
суммарный вес ≤ W, а суммарная ценность — максимальна.

Рекуррентное соотношение
Обозначим dp[i][j] — максимальная ценность, используя первые i предметов и вместимость j.

dp[0][j] = 0

dp[i][j] =
dp[i−1][j],
max(dp[i−1][j],
dp[i−1][j−w[i]] + v[i]),

если j < w[i]
если j ≥ w[i]

Объяснение:
•

либо предмет i не берём;

•

либо берём его и добавляем его ценность к оптимальному решению для остаточной
вместимости. filecite turn10file1

Оптимизация по памяти
Из файла:
Можно хранить одномерный массив:
dp[j] — максимум для вместимости j
Переход:
for i = 1..n
for j = W down to w[i]
dp[j] = max(dp[j], dp[j − w[i]] + v[i])

Почему внутренний цикл идёт в обратном порядке
Обоснование из файла:
•

если идти вперёд, dp[j − w[i]] уже может быть обновлён на текущем i;

•

это приведёт к многократному использованию одного и того же предмета;

•

обратный порядок гарантирует, что каждый предмет учитывается ровно один раз.

Динамическое программирование по подмножествам
Определение (из файла):
Динамическое программирование по подмножествам применяется, когда состояние определяется
множеством уже выбранных элементов, обычно кодируемым битовой маской.
Состояние имеет вид:
dp[mask][v]
где mask — подмножество элементов, v — дополнительный параметр (например, текущая
вершина).

Пример: задача коммивояжёра (TSP)
Формулировка (из файла):
Найти кратчайший путь, проходящий через все вершины графа ровно один раз и возвращающийся
в начальную.
DP-состояние:
dp[mask][v] — минимальная длина пути, который:
•

посещает все вершины из mask,

•

заканчивается в вершине v.

Переход (идея):
dp[mask][v] = min over u in mask\{v} (
dp[mask without v][u] + dist[u][v]
)
Сложность:
O(n^2 · 2^n) по времени и O(n · 2^n) по памяти.

dp[row][mask]
где mask кодирует занятость ячеек в текущей строке.

Задача (из файла):

•

•

Дерево отрезков
Определение (из файла):

дерево хранится в массиве t;

•

для вершины с индексом v:

–

левый сын: 2·v

–

правый сын: 2·v + 1

•

корень хранится в t[1];

•

размер массива — порядка 4·n, что достаточно для хранения всего дерева.

18. Как в дереве отрезков реализованы операции запроса на отрезке и обновления
элемента? В чём заключается идея «отложенных обновлений» в дереве отрезков? В
каких случаях она необходима?

Запрос на отрезке и обновление элемента в дереве отрезков
Запрос на отрезке (из файла):
Рекурсивно спускаемся по дереву. Для узла с отрезком [l, r] и запроса [ql, qr]:
•

если отрезки не пересекаются — возвращаем нейтральный элемент;

•

если [l, r] целиком внутри [ql, qr] — возвращаем значение узла;

•

иначе — делим запрос на детей и объединяем ответы.

Псевдокод
query(v, l, r, ql, qr)
if r < ql or qr < l
return NEUTRAL
if ql ≤ l and r ≤ qr
return t[v]
mid ← (l + r) div 2
return combine(
query(2v, l, mid, ql, qr),
query(2v+1, mid+1, r, ql, qr)
)
Сложность: O(log n) — посещается ограниченное число вершин.
Обновление одного элемента (из файла):
Спускаемся к листу, соответствующему позиции pos, меняем значение и пересчитываем путь
вверх.
Псевдокод
update(v, l, r, pos, new_val)
if l = r
t[v] ← new_val
return
mid ← (l + r) div 2
if pos ≤ mid
update(2v, l, mid, pos, new_val)
else
update(2v+1, mid+1, r, pos, new_val)
t[v] ← combine(t[2v], t[2v+1])
Сложность: O(log n).

Отложенные обновления (lazy propagation)
Идея (из файла):
При обновлении целого отрезка изменение не проталкивается сразу к листьям.
В узле сохраняется «отложенное» обновление, которое будет применено к детям только при
необходимости (перед следующим спуском).
Используется дополнительный массив lazy.
Принцип:
•

если обновляемый отрезок полностью покрывает узел — обновляем t[v] и записываем
пометку в lazy[v];

•

при дальнейшем спуске пометка «проталкивается» к детям.

Когда это необходимо
Из файла:
•

при массовых обновлениях на отрезках (прибавить, присвоить и т.п.);

•

без отложенных обновлений каждое такое обновление стоило бы O(n);

•

с lazy propagation и запросы, и обновления работают за O(log n).

Лекция 10. Дерево Фенвика. Разреженная таблица.
19. Что такое «дерево Фенвика»? Какие операции оно поддерживает и в чём его
преимущества перед деревом отрезков? Объясните, как работает операция
вычисления суммы на префиксе в дереве Фенвика, используя операцию & i i− .

Дерево Фенвика (Binary Indexed Tree)
Определение (из файла):
Дерево Фенвика — это структура данных для работы с массивом, позволяющая эффективно
считать префиксные суммы и выполнять точечные обновления.

Поддерживаемые операции и их сложность
Из файла:
•

sum(i) — сумма элементов на префиксе [1..i] — O(log n)

•

add(i, x) — прибавить x к элементу с индексом i — O(log n)

•

построение дерева — **O(n log n)** (или O(n)` специальным способом)

Преимущества перед деревом отрезков
Из файла:
•

проще в реализации;

•

требует O(n) памяти (против ~4n у дерева отрезков);

•

меньшая константа во времени;

•

эффективно работает именно для префиксных сумм.

(Отмечено, что дерево Фенвика поддерживает не все операции, в отличие от дерева отрезков.)

Структура и идея хранения
Из файла:
Используется массив t[1..n], где
t[i] хранит сумму элементов на отрезке
(i − (i & −i) + 1) .. i

Операция вычисления суммы на префиксе и роль i & -i
Идея (из файла):
Операция i & -i выделяет младший установленный бит числа i.
Он определяет длину отрезка, за который отвечает t[i].
Псевдокод вычисления префиксной суммы
sum(i)
res ← 0
while i > 0
res ← res + t[i]
i ← i − (i & −i)
return res
Объяснение:
На каждом шаге мы переходим к следующему узлу, который хранит сумму для предыдущего
блока элементов, постепенно покрывая весь префикс [1..i].

20. Для каких задач предназначена разреженная таблица? Каковы её ограничения?
Опишите процесс предобработки и выполнения запроса в разреженной таблице для
задачи поиска минимума на отрезке. Почему операция запроса выполняется за O(1)
неприменима для обновлений?

Разреженная таблица (Sparse Table)
Назначение (из файла):
Разреженная таблица применяется для задач поиска минимума на отрезке (RMQ) в статическом
массиве, то есть без обновлений. filecite turn16file0

Ограничения
Из файла:
•

массив не должен изменяться;

•

структура не поддерживает обновления;

•

подходит только для идемпотентных операций (например, min, max).

Предобработка
Идея (из файла):
Строится таблица:
st[i][k] — минимум на отрезке длины 2^k,
начинающемся в позиции i
База
st[i][0] = a[i]

Переход
st[i][k] = min(st[i][k−1],
st[i + 2^(k−1)][k−1])
Сложность предобработки:
O(n log n) по времени и памяти.

Запрос минимума на отрезке
Для запроса [l, r]:
•

k = floor(log2(r − l + 1))

•

ответ:

min(st[l][k],
st[r − 2^k + 1][k])
Сложность запроса: O(1).

Почему запрос O(1), но нет обновлений
Из лекции:
•

минимум на отрезке можно получить из двух заранее посчитанных блоков;

•

при изменении одного элемента нужно пересчитать множество значений таблицы,
что требует O(n log n);

•

поэтому разреженная таблица неприменима для обновлений.

Лекция 11. Сбалансированные деревья поиска.
21. Что такое «сбалансированное дерево поиска» и зачем оно нужно? В чём проблема
несбалансированного двоичного дерева поиска?

Сбалансированное дерево поиска
Определение (из файла):
Сбалансированное дерево поиска — это двоичное дерево поиска, у которого высота
поддерживается логарифмической за счёт специальных правил балансировки.
Зачем нужно:
Оно обеспечивает выполнение основных операций поиска, вставки и удаления за
O(log n).

Проблема несбалансированного двоичного дерева поиска
Суть проблемы:
В обычном двоичном дереве поиска при неудачном порядке вставки дерево может
выродиться в цепочку.
В этом случае:
•

высота дерева становится O(n);

•

операции search, insert, delete работают за O(n) вместо O(log n).

Вывод
•

несбалансированное ДДП не гарантирует логарифмическую сложность;

•

сбалансированные деревья устраняют вырождение и гарантируют O(log n) для операций.

22. Опишите инвариант АВЛ-дерева. Какие повороты используются для
поддержания его сбалансированности? Опишите основную идею Сплей-дерева. Что
такое операция подъёма к корню и какие шаги она включает?

АВЛ-дерево: инвариант и повороты
Инвариант АВЛ-дерева (из файла):
Для любой вершины высоты её левого и правого поддеревьев отличаются не более чем на 1:
|ℎ(𝑙𝑒𝑓𝑡) − ℎ(𝑟𝑖𝑔ℎ𝑡)| ≤ 1
Это гарантирует, что высота дерева равна O(log n).

Повороты в АВЛ-дереве
Используемые повороты (из файла):
1.

Малый правый поворот (LL)

2.

Малый левый поворот (RR)

3.

Большой левый-правый (LR)

4.

Большой правый-левый (RL)

Выбор поворота зависит от того, в каком поддереве произошло нарушение баланса.

Сплей-дерево: основная идея
Определение (из файла):
Сплей-дерево — это двоичное дерево поиска, которое самоадаптируется:
каждый раз при обращении к вершине она поднимается в корень.
Баланс явно не хранится, но за счёт перестроений достигается хорошая амортизированная
асимптотика.

Операция подъёма к корню (splay)
Суть (из файла):
Операция splay(x) поднимает вершину x в корень дерева серией поворотов.

Основные шаги (случаи):

1.

Zig
x — ребёнок корня → один поворот

2.

Zig-Zig
x и его родитель — оба левые или оба правые дети → два однотипных поворота

3.

Zig-Zag
x — левый ребёнок, родитель — правый (или наоборот) → два разных поворота

Лекция 12. Декартово дерево по неявному ключу.
24. Что такое декартово дерево? Какие два свойства оно сочетает?

Декартово дерево (treap)
Определение (из файла):
Декартово дерево — это структура данных, которая является двоичным деревом поиска по
ключам и одновременно кучей по приоритетам.

Какие два свойства оно сочетает
Из лекциям:
1.

Свойство двоичного дерева поиска (BST) — по ключу x:

•

все ключи в левом поддереве меньше x;

•

все ключи в правом поддереве больше x.

2.

Свойство кучи (heap) — по приоритету y:

•

приоритет в вершине не больше (или не меньше — в зависимости от варианта) приоритетов
в её детях
(обычно — min-heap по y).

Небольшая реализация декартова дерева по неявному ключу (Treap) в псевдокоде (ключевые
операции merge, split, insert, delete) — как в файле.

text
# Вспомогательные
size(node):
return node.size if node != null else 0
update_size(node):
node.size = 1 + size(node.left) + size(node.right)
# merge(A, B): все элементы A должны быть "левее" всех элементов B
merge(A, B):
if A == null: return B
if B == null: return A
if A.priority > B.priority:
A.right = merge(A.right, B)
update_size(A)
return A
else:
B.left = merge(A, B.left)
update_size(B)
return B
# split(T, k): L содержит первые k элементов, R — остальные

split(T, k):
if T == null: return (null, null)
left_size = size(T.left)
if k <= left_size:
(L, R) = split(T.left, k)
T.left = R
update_size(T)
return (L, T)
else:
(L, R) = split(T.right, k - left_size - 1)
T.right = L
update_size(T)
return (T, R)
# Вставка по позиции pos (0-index)
insert(T, pos, value, priority):
(L, R) = split(T, pos)
new_node = Node(value, priority)
return merge(merge(L, new_node), R)
# Удаление элемента на позиции pos
delete(T, pos):
(L, R) = split(T, pos)
(M, R2) = split(R, 1)
# M — удаляемый узел
return merge(L, R2)

25. Объясните операции слияния и разделения в декартовом дереве. Какова их

использоваться для работы с последовательностями? Приведите примеры

Операции слияния и разделения в декартовом дереве
Слияние (merge)
Идея (из файла):
merge(A, B) объединяет два декартовых дерева, при этом предполагается, что все ключи A
меньше всех ключей B.
Корнем результата становится вершина с большим приоритетом, а слияние рекурсивно
продолжается в соответствующем поддереве.
Псевдокод (как в файле):
merge(A, B)
if A = null return B
if B = null return A
if A.priority > B.priority
A.right = merge(A.right, B)
return A
else
B.left = merge(A, B.left)
return B

Разделение (split)
Идея (из файла):
split(T, x) делит дерево T на два:
•

L — все элементы с ключами < x

•

R — все элементы с ключами ≥ x

Разделение идёт рекурсивно, сравнивая x с ключом текущей вершины.
Псевдокод:
split(T, x)
if T = null
return (null, null)
if key(T) < x
(L, R) = split(T.right, x)
T.right = L
return (T, R)
else
(L, R) = split(T.left, x)
T.left = R
return (L, T)

Временная сложность
Из файла:
•

merge — O(h)

•

split — O(h)

где h — высота декартова дерева.
Так как приоритеты выбираются случайно, математическое ожидание высоты равно O(log n),
значит:

merge, split — O(log n) в среднем

Декартово дерево с неявным ключом
Идея (из файла):
Ключ явно не хранится.
Позиция элемента в последовательности определяется:

key = size(left_subtree)
То есть порядком обхода in-order.
В вершине дополнительно хранится size поддерева.

Использование для работы с последовательностями
Из файла:
С неявным ключом декартово дерево позволяет работать с последовательностью как с массивом,
поддерживая операции за O(log n):
Примеры операций:
•

вставка элемента в позицию i;

•

удаление элемента по позиции;

•

получение элемента по индексу;

•

вырезание подотрезка [l, r];

•

вставка подотрезка в другое место;

•

разворот подотрезка (с доп. флагом).

Все они реализуются через комбинации split и merge.

26. Сравните декартово дерево с другими структурами данных, например, с деревом

Декартово дерево сочетает BST по ключу и кучу по приоритету; при случайных приоритетах имеет

•

менее гибко для операций со структурой массива

Лекция 13. Наименьший общий предок.
27. Дайте определение наименьшего общего предка двух узлов в дереве. Приведите
пример. Опишите метод бинарных подъёмов. Какая предобработка требуется и

Наименьший общий предок (LCA)
Определение (из файла):
Наименьший общий предок двух вершин u и v в дереве — это вершина наибольшей глубины,
которая является предком и u, и v.
Пример:
Если вершина x — предок u и v, и никакой её потомок уже не является общим предком, то x —
LCA(u, v).

Метод бинарных подъёмов
Идея (из файла):
Для каждой вершины заранее хранятся её предки на расстояниях
20 , 21 , 22 , … Это позволяет «поднимать» вершину вверх по дереву скачками степеней двойки.

Предобработка
Из файла:
1.

Выполняется DFS/BFS от корня.

2.

Для каждой вершины v сохраняется:
–

up[v][0] — её непосредственный родитель;

–

up[v][k] — предок на расстоянии 2^k.

Рекуррентно:
up[v][k] = up[ up[v][k−1] ][k−1]
Также сохраняется глубина depth[v].
Сложность предобработки:
𝑂(𝑛log𝑛)

Запрос LCA(u, v)
Из файла:
1.

Если depth[u] < depth[v], поменять их местами.

2.

Поднять u до уровня v, используя бинарные подъёмы.

3.

Если u = v, это и есть LCA.

4.

Иначе одновременно поднимать u и v от больших k к меньшим,
пока их предки различны.

5.

Ответ — parent[u].

Сложность запроса
Результат (из файла):
•

время одного запроса LCA — O(log n);

•

память — O(n log n).

Идея (из файла):
Задача нахождения наименьшего общего предка двух вершин в дереве сводится к задаче поиска

Эйлеров обход
Из файла:
Выполняется DFS, при котором:
•

при каждом посещении вершины она добавляется в массив euler;

•

параллельно сохраняется массив depth — глубина каждой записи в euler;

•

для каждой вершины v запоминается first[v] — позиция первого появления v в euler.

Длина массива euler равна 2n − 1.

Как получается НОП через RMQ
Из файла:
Для двух вершин u и v:
1.

берём их первые вхождения l = first[u], r = first[v];

2.

рассматриваем отрезок euler[l..r];

3.

вершина с минимальной глубиной на этом отрезке и есть LCA(u, v).

Таким образом:
LCA(u, v) = argmin depth[i] for i in [first[u], first[v]]

Алгоритм с разреженной таблицей
Из файла:

Выполнить эйлеров обход — O(n)

2.

Построить разреженную таблицу по массиву depth — O(n log n)

3.

Каждый запрос RMQ (и, значит, LCA) — O(1)

Итог:
•

предобработка: O(n log n)

•

запрос LCA: O(1)

•

память: O(n log n)

29. Что такое Heavy-Light декомпозиция? Каковы её ключевые понятия? Опишите
шаги построения Heavy-Light декомпозиции. Какие свойства дерева используются?

Heavy-Light декомпозиция (HLD)
Определение (из файла):
Heavy-Light декомпозиция — это способ разбиения дерева на пути, позволяющий свести запросы
на пути между вершинами к логарифмическому числу запросов на отрезках.

Ключевые понятия
Из файла:
•

Размер поддерева sz[v] — число вершин в поддереве вершины v.

•

Тяжёлое ребро (heavy edge) — ребро (v, u), где u — сын v с максимальным поддеревом.

•

Лёгкое ребро (light edge) — любое другое ребро.

•

Тяжёлый путь — последовательность вершин, соединённых тяжёлыми рёбрами.

Каждая вершина принадлежит ровно одному тяжёлому пути.

Шаги построения Heavy-Light декомпозиции
Из файла:
1.

2.

3.

DFS от корня
–

вычисляются sz[v] — размеры поддеревьев;

–

для каждой вершины выбирается её тяжёлый сын.

Разбиение на пути
–

тяжёлые рёбра образуют тяжёлые пути;

–

при переходе по лёгкому ребру начинается новый путь.

Нумерация вершин
–

вершины каждого тяжёлого пути нумеруются подряд;

–

каждому пути сопоставляется отрезок в массиве.

После этого запросы на пути в дереве сводятся к запросам на отрезках массива.

Используемые свойства дерева
Из файла:
•

у каждой вершины не более одного тяжёлого сына;

•

при переходе по лёгкому ребру размер поддерева уменьшается как минимум в 2 раза;

•

на любом пути от вершины к корню встречается не более O(log n) лёгких рёбер.

Отсюда следует, что любой путь в дереве разбивается на O(log n) тяжёлых путей.

Использование Heavy-Light декомпозиции (HLD) для запросов на путях
Идея (из файла):
HLD позволяет разбить путь между двумя вершинами в дереве на O(log n) непересекающихся
отрезков, каждый из которых соответствует непрерывному отрезку в линейном массиве. Тогда
запрос на пути сводится к нескольким запросам на отрезках.

Как выполняется запрос на пути u — v
Из файла:
1.

2.

Пока вершины u и v находятся в разных тяжёлых путях:
–

выбираем вершину с большей глубиной головы пути;

–

добавляем запрос на отрезке от этой вершины до головы её пути;

–

поднимаем вершину к родителю головы пути (переход по лёгкому ребру).

Когда u и v оказываются в одном тяжёлом пути:

Все запросы объединяются (суммируются, берётся минимум и т.п.).

Пример (из файла)
Задача:
Найти сумму значений вершин на пути между u и v.
Решение с HLD:
•

после HLD каждой вершине сопоставлен индекс в массиве;

•

над массивом построено дерево отрезков;

•

путь u — v разбивается на O(log n) отрезков массива;

число отрезков на пути — O(log n);

каждый запрос на отрезке — O(log n);

31. Что такое центроидная декомпозиция? Как находится центроид? Опишите
алгоритм построения центроидной декомпозиции и сравните её с HLD.

Центроидная декомпозиция
Определение (из файла):
Центроидная декомпозиция — это рекурсивное разбиение дерева по центроидам, при котором
строится новое дерево (дерево центроидов), отражающее иерархию разбиений.

Что такое центроид и как он находится
Центроид дерева (из файла):
Центроид — это вершина, удаление которой разбивает дерево на компоненты, каждая из которых
имеет размер не более n/2, где n — размер текущего дерева.
Как найти центроид:
1.

выполнить DFS и посчитать размеры поддеревьев;

2.

для каждой вершины проверить, что размер каждого поддерева ≤ n/2;

3.

такая вершина и есть центроид (он существует и может быть не единственным).

Алгоритм построения центроидной декомпозиции
Из файла:
1.

Найти центроид текущего дерева.

2.

Сделать его корнем центроидного дерева.

3.

Удалить центроид, получив несколько компонент.

4.

Рекурсивно применить алгоритм к каждой компоненте.

Рекурсия продолжается, пока компонента не станет пустой.
Глубина центроидного дерева: O(log n).

Сравнение центроидной декомпозиции и HLD
Центроидная декомпозиция:
•

ориентирована на задачи со всеми расстояниями до поддеревьев или глобальные запросы;

•

дерево делится по вершинам, а не по путям;

•

глубина разбиения — O(log n);

•

часто используется для задач вида «для каждой вершины найти что-то относительно всех
остальных».

Heavy-Light декомпозиция (HLD):
•

ориентирована на запросы на путях;

•

разбивает дерево на тяжёлые пути;

•

путь между вершинами делится на O(log n) отрезков;

•

хорошо сочетается с деревом отрезков.

Ключевое различие (из файла)
•

HLD — для запросов на путях (u–v).

•

Центроидная декомпозиция — для запросов, зависящих от расстояний и поддеревьев
относительно центра.

32. Опишите алгоритм обхода в глубину для графа. Какова его сложность? Как с
помощью обхода в глубину найти цикл в неориентированном и ориентированном
графах?

Обход в глубину (DFS)
Определение (из файла):
Обход в глубину — это алгоритм обхода графа, при котором из текущей вершины рекурсивно
посещается любой ещё не посещённый сосед, пока это возможно, после чего происходит возврат
назад.

Алгоритм DFS
Псевдокод
dfs(v)
used[v] ← true
for each u in adj[v]
if not used[u]
dfs(u)
Обход запускается для всех вершин (если граф несвязный).

Сложность DFS
Из файла:
•

по времени: O(n + m)
где n — число вершин, m — число рёбер;

•

по памяти: O(n) (массив посещённых вершин + стек рекурсии).

Поиск цикла в неориентированном графе
Идея (из файла):
Во время DFS для каждой вершины запоминается её родитель.
Если из вершины v есть ребро в уже посещённую вершину u,
и u не является родителем v, то найден цикл.
Условие цикла:
used[u] = true and u ≠ parent[v]

Поиск цикла в ориентированном графе
Идея (из файла):
Используется раскраска вершин:
•

0 — не посещена

•

1 — в текущем стеке рекурсии

•

2 — обработана полностью

Псевдокод
dfs(v)
color[v] ← 1
for each u in adj[v]
if color[u] = 0
dfs(u)
else if color[u] = 1
cycle found
color[v] ← 2
Пояснение:
Ребро в вершину с цветом 1 — это обратное ребро, оно образует цикл.

Кратко:
•

DFS обходит граф, углубляясь максимально глубоко

•

сложность: O(n + m)

•

цикл в неориентированном графе — ребро к посещённой вершине, не являющейся
родителем

•

цикл в ориентированном графе — обратное ребро в вершину из стека рекурсии

33. Что такое топологическая сортировка? Приведите алгоритм Кана и алгоритм на
основе обхода в глубину. Примеры применения.

Топологическая сортировка
Определение (из файла):
Топологическая сортировка — это упорядочивание вершин ориентированного ациклического
графа (DAG) такое, что для каждого ребра u → v вершина u идёт раньше v.

Алгоритм Кана (по входящим степеням)
Идея (из файла):
Вершины с нулевой входящей степенью можно ставить первыми; после их удаления степени
пересчитываются.
Псевдокод
kahn_toposort(G)
for v in V
indeg[v] ← number of incoming edges
Q ← all v with indeg[v] = 0
order ← empty list
while Q not empty
v ← pop(Q)
add v to order
for each (v → u)
indeg[u] ← indeg[u] − 1
if indeg[u] = 0
push u to Q
if |order| < |V|
cycle exists
return order
Сложность: O(n + m).

Алгоритм на основе DFS
Идея (из файла):
Вершина добавляется в ответ после обработки всех её потомков; итоговый порядок — обратный
времени выхода.
Псевдокод
dfs(v)
used[v] ← true
for each (v → u)
if not used[u]
dfs(u)
add v to order
toposort_dfs(G)
for v in V
if not used[v]
dfs(v)
reverse(order)
Сложность: O(n + m).

Примеры применения
•

планирование задач с зависимостями;

•

порядок компиляции модулей;

•

анализ зависимостей пакетов;

•

упорядочивание шагов вычислений в DAG.

Важно: топологическая сортировка существует только для DAG; наличие цикла означает, что
сортировка невозможна.

34. Дайте определение компоненты сильной связности. Опишите алгоритм

Компонента сильной связности (КСС)
Определение (из файла):
Компонента сильной связности ориентированного графа — это максимальное множество
вершин, такое что для любых двух вершин u и v из этого множества существуют пути u → v и v →
u.

Алгоритм Косарайю
Идея (из файла):
Алгоритм использует два обхода в глубину и обращение рёбер графа.

Шаги алгоритма:
1.

Выполнить DFS в исходном графе и записать вершины в порядке убывания времени выхода
(в стек или список).

2.

Построить транспонированный граф (все рёбра обратить).

3.

Запускать DFS в транспонированном графе, извлекая вершины в порядке из шага 1.
Все вершины, посещённые за один DFS, образуют одну компоненту сильной связности.

Алгоритм выполняет два DFS ⇒ O(n + m).

Определение (из файла):
Конденсация графа — это граф, полученный из исходного сжатием каждой компоненты сильной
связности в одну вершину.
Свойства:
•

конденсация — ориентированный ациклический граф (DAG);

•

между двумя вершинами конденсации есть ребро, если в исходном графе было ребро между
соответствующими КСС.

35. Дайте определение моста и точки сочленения. Опишите алгоритм их поиска. Что
такое Эйлеров цикл? Каковы условия его существования? Опишите алгоритм
нахождения Эйлерова цикла.

Мост и точка сочленения
Определения (из файла):
•

Мост — ребро неориентированного графа, удаление которого увеличивает число компонент
связности.

•

Точка сочленения — вершина, удаление которой (вместе с инцидентными рёбрами)
увеличивает число компонент связности.

Алгоритм поиска мостов и точек сочленения (DFS)
Идея (из файла):
Используется обход в глубину с временами входа tin[v] и значениями low[v] — минимальное
tin, достижимое из v по рёбрам дерева DFS и одному обратному ребру.

Правила:
•

Ребро (v, u) — мост, если
low[u] > tin[v]

•

Вершина v — точка сочленения, если:
–

v не корень DFS и существует ребёнок u с
low[u] ≥ tin[v]

–

v — корень DFS и имеет более одного ребёнка в дереве DFS. filecite turn31file1

Сложность: O(n + m).

Эйлеров цикл
Определение (из файла):
Эйлеров цикл — это цикл в графе, проходящий по каждому ребру ровно один раз и
возвращающийся в исходную вершину. filecite turn31file2

Условия существования Эйлерова цикла
Для неориентированного графа (из файла):
•

граф связен (если игнорировать изолированные вершины);

•

степени всех вершин чётные.

Для ориентированного графа (из файла):
•

для каждой вершины in-degree = out-degree;

•

все вершины с ненулевой степенью лежат в одной компоненте сильной связности.

Алгоритм нахождения Эйлерова цикла
Алгоритм Хиерхольцера (из файла):
Идея:
Пока есть неиспользованные рёбра, идём по ним, удаляя из графа, формируя цикл; при встрече
вершины с неиспользованными рёбрами «вклеиваем» новый цикл.
Псевдокод (схема):
stack ← [start]
path ← empty
while stack not empty
v ← top(stack)
if v has unused edges
take one (v,u), mark used
push u to stack
else
pop v from stack
add v to path
Полученный path (в обратном порядке) — эйлеров цикл.
Сложность: O(n + m).

•

union — объединить множества

Используется в алгоритме Крускала.
Шаги:
1.

отсортировать рёбра по весу;

2.

3.

Сложность:

Алгоритм Прима
Идея (из файла):
Строит МОД, расширяя текущее дерево, каждый раз добавляя минимальное ребро,

Шаги:
1.

выбрать стартовую вершину;

2.

поддерживать для каждой вершины минимальное ребро к текущему дереву;

3.

на каждом шаге добавлять вершину с минимальным таким ребром.

Сложность:
•

с приоритетной очередью: O(m log n);

•

с матрицей смежности: O(n^2).

Метод цепочек (chaining)
Суть (из файла):
В каждой ячейке таблицы хранится цепочка (список) элементов с одинаковым хешем.

дополнительная память на списки;

Метод открытой адресации

усложнённое удаление;

•

•

Открытая адресация — быстрее и компактнее при низком коэффициенте заполнения, но

40. Что такое идеальное хеширование? Опишите двухуровневую схему идеального
хеширования. Что такое хеширование кукушки? Как выполняются вставка и поиск?

Идеальное хеширование
Определение (из файла):
Идеальное хеширование — это метод хеширования для статического множества ключей, при
котором коллизии отсутствуют, а поиск выполняется за O(1) в худшем случае.

Двухуровневая схема идеального хеширования
Идея (из файла):
Используются две хеш-таблицы и две хеш-функции.

Первый уровень
•

выбирается хеш-функция h₁;

•

все ключи распределяются по ячейкам основной таблицы;

•

в каждой ячейке образуется свой набор ключей.

Второй уровень
•

для каждой ячейки с k ключами строится своя таблица размера k²;

•

подбирается хеш-функция h₂, при которой внутри ячейки нет коллизий;

•

если коллизии есть — h₂ выбирается заново.

Свойства (из файла):
•

суммарная память — O(n);

•

поиск — O(1) в худшем случае;

•

структура не поддерживает эффективные вставки (статическая).

Хеширование кукушки (Cuckoo hashing)
Идея (из файла):
Используются две хеш-функции h₁, h₂.
Каждый ключ может храниться в одной из двух позиций.

Поиск
Поиск (из файла):
Для ключа x проверяются две ячейки:
h₁(x) и h₂(x)
Если ключ есть — он обязательно в одной из них.
Сложность поиска: O(1) в худшем случае.

Вставка
Вставка (из файла):
1.

поместить ключ в позицию h₁(x);

2.

если ячейка занята — вытеснить старый ключ;

3.

вытесненный ключ попытаться вставить в его альтернативную позицию;

4.

процесс повторяется (как «кукушка выталкивает яйца»).

Если возникает цикл вытеснений, выполняется перехеширование с новыми хеш-функциями.

Сравнение (из файла)
•

•

Идеальное хеширование:
–

статическое множество;

–

поиск O(1) в худшем случае;

–

нет вставок.

Хеширование кукушки:
–

динамическое;

–

поиск O(1) в худшем случае;

–

вставка обычно O(1), но возможна перехешировка.

Статические члены класса
Статические члены — это члены класса, которые:
•

принадлежат самому классу, а не объекту;

•

существуют в единственном экземпляре;

•

доступны без создания объекта.

В C# объявляются с ключевым словом static.
К статическим членам относятся:
•

статические поля;

•

статические методы;

•

статические свойства.

Пример (словесно):
счётчик объектов, общая константа, вспомогательная функция.

Статический конструктор
Статический конструктор:
•

инициализирует статические поля класса;

•

вызывается автоматически один раз;

•

выполняется до первого использования класса;

•

не имеет параметров и модификаторов доступа.

В классе может быть только один статический конструктор.

this — ссылка на текущий экземпляр класса.
Используется для:
1.

обращения к полям и методам текущего объекта;

2.

устранения конфликта имён (параметр = поле);

3.

вызова другого конструктора того же класса.

Пример (словесно):
this.x — поле объекта,
this.Method() — метод текущего объекта.
⇒ В статических методах this недоступно, так как нет экземпляра класса.

Наследование
Наследование — это механизм ООП, при котором один класс (производный) получает поля и
методы другого класса (базового) и может:
•

использовать их;

•

расширять;

•

переопределять поведение.

Цель — повторное использование кода и логическое обобщение.

Как обозначается наследование в C#
В C# наследование задаётся через двоеточие ::

csharp
class Derived : Base
{
}

Какие члены наследуются
В производном классе наследуются:
•

public — наследуются и доступны

•

protected — наследуются и доступны в наследнике

•

internal — наследуются (в пределах сборки)

•

protected internal — наследуются

⇒ Не наследуются:
•

private члены базового класса

•

конструкторы базового класса (но они вызываются)

Запечатанный класс (sealed)
Запечатанный класс — это класс, от которого нельзя наследоваться.
В C# помечается ключевым словом:

csharp
sealed class MyClass
{
}
Назначение:
•

запретить расширение класса;

•

повысить безопасность;

•

иногда — оптимизация.

1.

csharp

⇒ Разрешено множественное наследование интерфейсов.
2.

Нельзя наследоваться от sealed класса

3.

Производный класс не имеет доступа к private членам базового

4.

Переопределять можно только virtual / abstract члены
(через override)

5.

Базовый класс должен быть доступен (по модификатору доступа)

•

•

Полиморфизм
Полиморфизм — это способность объектов разных классов обрабатываться через общий
базовый тип, при этом вызываться будет реализация конкретного объекта.

Виртуальный метод — метод базового класса, который может быть переопределён в

Перегрузка методов (method overloading)
Перегрузка методов — это объявление нескольких методов с одинаковым именем, но разными
сигнатурами (набором параметров).
Различаться могут:
•

количество параметров;

•

типы параметров;

•

порядок параметров.

⇒ Нельзя перегружать метод только по возвращаемому типу.
Зачем нужна:
•

удобство и читаемость API;

•

логически одинаковые операции с разными входными данными.

Анонимные типы
Анонимный тип — это тип без явного имени, создаваемый «на лету» с набором свойств.
В C# создаётся с помощью new { ... }.
Особенности:
•

свойства только для чтения;

•

тип доступен только в текущем контексте;

•

компилятор сам генерирует класс.

Где применяются анонимные типы
Основные случаи использования:
•

LINQ-запросы (проекции данных);

•

временные структуры для группировки данных;

•

передача нескольких значений без объявления отдельного класса.

Типичный сценарий:
выбрать несколько полей из объекта и работать с ними как с единым значением.

Интерфейс
Интерфейс — это контракт, который задаёт набор членов, которые класс обязан реализовать, не
предоставляя реализации (классически).
В C# объявляется ключевым словом interface.

Чем интерфейс отличается от абстрактного класса
Интерфейс:
•

описывает что класс должен уметь;

•

не хранит состояние (полей экземпляра);

•

не содержит конструкторов;

•

класс может реализовать несколько интерфейсов.

Абстрактный класс:
•

может содержать реализацию и поля;

•

может иметь конструкторы;

•

поддерживает только одиночное наследование.

Сколько интерфейсов может реализовать класс
Сколько угодно.
В C# разрешено множественное наследование интерфейсов:

csharp
class MyClass : IFirst, ISecond, IThird
{
}

Наследование интерфейсов
Интерфейс может наследоваться от других интерфейсов:

csharp
interface IA { }
interface IB { }
interface IC : IA, IB { }
Класс, реализующий IC, обязан реализовать все члены всех базовых интерфейсов.

Все ли члены интерфейса должны быть реализованы
•

Да, если класс не абстрактный

•

Нет, если класс объявлен abstract

Начиная с C# 8, интерфейсы могут содержать реализации по умолчанию, но:
•

класс не обязан их переопределять;

•

это исключение, а не классическое правило.

Модификаторы по умолчанию
В интерфейсе:
•

все члены по умолчанию public;

•

указывать public необязательно;

•

private, protected (классически) недопустимы
(кроме новых возможностей C# 8+ для служебных методов).

Исключение
Исключение — это объект, описывающий ошибку во время выполнения программы, которая
нарушает нормальный ход выполнения.

Зачем нужна обработка исключений
Обработка исключений позволяет:
•

перехватывать ошибки без аварийного завершения программы;

•

корректно освобождать ресурсы;

•

отделять логику обработки ошибок от основной логики.

try — catch — finally

csharp
try
{

// код, где может возникнуть исключение
}
catch (ExceptionType e)
{
// обработка исключения
}
finally
{
// код очистки ресурсов
}
•

try — код с потенциальной ошибкой

•

catch — перехват и обработка исключения

•

finally — выполняется для завершения/очистки

Всегда ли выполняется finally?
Почти всегда:
•

выполняется при нормальном завершении try;

•

выполняется при возникновении исключения;

•

выполняется даже при return из try или catch.

Может не выполниться, если:
•

процесс аварийно завершён;

•

вызван Environment.FailFast.
