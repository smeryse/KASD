# Множественный поиск подстрок (Ахо-Корасик)

Сначала про общую идею, на которой всё держится. Здесь используется алгоритм Ахо-Корасик для одновременного поиска множества паттернов в тексте. Ключевая мысль в том, что можно построить конечный автомат по набору паттернов, который за один проход по тексту найдёт все вхождения всех паттернов. Автомат строится на основе бора (префиксного дерева) с добавлением failure links — переходов, которые позволяют продолжить поиск при несовпадении, не возвращаясь к началу.

Первый логический блок — ввод данных в Solve. Этот блок решает задачу "прочитать количество паттернов n, сами паттерны и текст t". Он нужен, потому что без входных данных нечего обрабатывать. Взаимодействие здесь простое: FastScanner читает число n, затем n строк в массив patterns, затем текст t.

Второй логический блок — класс AhoCorasick и его конструктор. Этот блок решает задачу "построить автомат Ахо-Корасик по набору паттернов". Он нужен, чтобы подготовить структуру данных для эффективного поиска. Конструктор работает в два этапа:

**Этап 1 — построение бора.** Бор — это дерево, где каждый узел соответствует префиксу некоторого паттерна. Корень (узел 0) представляет пустой префикс. Для каждого паттерна проходим по бору: если переход по очередному символу ещё не создан, создаём новый узел. В конце паттерна добавляем его индекс в output текущего узла. output[u] хранит список индексов паттернов, заканчивающихся в узле u. Взаимодействие: trie[node].next — словарь переходов, trie[node].output — список завершающихся паттернов.

**Этап 2 — построение failure links через BFS.** Failure link fail[u] указывает на узел, соответствующий наибольшему суффиксу пути к u, который также является префиксом некоторого паттерна. Это позволяет при несовпадении "перепрыгнуть" на более короткий префикс, не начиная с корня. Алгоритм: начинаем BFS с детей корня (их fail = 0). Для каждого узла u и символа c, если есть переход v = trie[u].next[c], то fail[v] вычисляется так: берём fail[u] и идём по failure links, пока не найдём узел, имеющий переход по c, или не дойдём до корня. Также добавляем в output[v] все output из fail[v] — это нужно, чтобы не пропустить паттерны, являющиеся суффиксами других паттернов. Взаимодействие: queue обеспечивает обход в ширину, trie[u].fail хранит failure link.

Третий логический блок — метод Search в AhoCorasick. Этот блок решает задачу "найти все паттерны, встречающиеся в тексте t". Он нужен для получения ответа. Как он работает: начинаем в корне (node = 0), для каждого символа текста t:

1. Пока нет перехода по текущему символу и мы не в корне, идём по failure link (node = trie[node].fail).
2. Если переход есть, переходим по нему.
3. Для всех паттернов в output текущего узла помечаем их как найденные (found[idx] = true).

Важно: output уже содержит все паттерны, заканчивающиеся в текущей позиции, включая те, что пришли из failure links. Взаимодействие: метод возвращает массив bool, где found[i] = true, если паттерн i найден.

Четвёртый логический блок — вывод результата в Solve. Этот блок решает задачу "вывести YES/NO для каждого паттерна в порядке ввода". Он нужен для соответствия формату вывода. Взаимодействие: ac.Search(t) возвращает массив found, затем для каждого i от 0 до n-1 выводим "YES" или "NO" с новой строки.

Пятый логический блок — FastScanner. Этот блок решает задачу "быстрый ввод данных". Он нужен для эффективного чтения больших объёмов данных (сумма длин паттернов до 10^6, текст до 10^6). NextInt() читает целое число, NextString() читает строку.

Шестой логический блок — Program.Main. Его задача — просто запустить решение: вызвать MultipleSearch.Solve(). Он нужен как точка входа приложения C#.

Итоговое резюме. Программа читает n паттернов и текст t, строит автомат Ахо-Корасик и находит все паттерны, встречающиеся в t. Бор строится за O(сумма длин паттернов), failure links — за то же время через BFS, поиск по тексту — за O(|t|). Общая сложность линейна по сумме длин всех строк. Для каждого паттерна выводится YES, если он найден, и NO иначе.
